## FRAMEWORK 1 — GTD (David Allen)

**핵심 원칙:** "머릿속을 비워라. 시스템이 기억하게 하라."

1인 사업자의 가장 큰 적은 **머릿속에 쌓이는 할 일들**.
GTD는 이걸 꺼내서 신뢰할 수 있는 시스템에 넣는 방법론.

### GTD 5단계 → JW Financial 적용

```
1. 수집 (Capture)
   머릿속, 카톡, 텔레그램, 이메일에 흩어진 모든 것을 한 곳으로
   → JW Financial 수집함: Telegram 봇 (언제든 보내면 Notion에 쌓임)

2. 명확화 (Clarify)
   이게 할 수 있는 일인가? 다음 행동이 무엇인가?
   → "고객 팔로업 필요" (X) → "홍길동 D+7 카카오 메시지 보내기" (O)

3. 정리 (Organize)
   프로젝트 / 다음 행동 / 언젠가 / 대기중으로 분류
   → Notion 4가지 뷰로 관리

4. 검토 (Reflect)
   주간 리뷰: 매주 금요일 30분
   → 완료 확인, 다음 주 우선순위 세팅, 고객 팔로업 누락 체크

5. 실행 (Engage)
   지금 상황(장소/에너지/시간)에 맞는 것 실행
   → 이동 중: 카카오 메시지 발송
   → 집중 시간: 콘텐츠 작성, 포트폴리오 북
   → 짜투리: 팔로업 메시지 확인
```

### JW Financial 주간 리뷰 체크리스트

```
매주 금요일 (30분)

[ ] 이번 주 미팅 완료 고객 → 다음 액션 설정
[ ] D+1/D+2/D+3 대기 고객 메시지 발송 확인
[ ] D+7/D+30/D+90 도래 고객 확인 및 메시지 준비
[ ] 상담 신청 → 확정 → 미진행 건 체크
[ ] 콘텐츠 발행 현황 확인 (이번 주 발행 / 다음 주 예정)
[ ] 핀사이트랩스 다음 주 리포트 주제 세팅
[ ] 레포 + Notion 업데이트 필요 항목 확인
```

## FRAMEWORK 2 — 도요타 생산방식 (Taiichi Ohno)

**핵심 원칙:** "가치를 만들지 않는 모든 것은 낭비다. 낭비를 제거하라."

도요타의 7가지 낭비를 JW Financial 1인 사무소에 적용:

```
① 과잉생산 낭비
   → 보내지도 않을 콘텐츠 초안, 쓰지도 않을 자료 과잉 생산
   → 해결: 요청받은 것만, 바로 쓸 것만 만들기

② 대기 낭비
   → 고객 답장 기다리는 동안 아무것도 못 하는 상태
   → 해결: 대기 중 할 수 있는 짧은 작업 목록 항상 준비

③ 이동 낭비
   → 정보가 카톡/텔레그램/이메일/노션에 흩어져 매번 찾아야 함
   → 해결: Telegram 봇 → Notion 단일 수렴 구조 강화

④ 재작업 낭비
   → 초안 → 피드백 → 재작업 반복
   → 해결: 작업 전 요구사항 명확화 (부장 지시 시 구체적 조건 명시)

⑤ 재고 낭비
   → 작성했지만 발송 안 한 메시지, 만들었지만 안 쓴 자료
   → 해결: 만들면 바로 쓰거나 폐기, 재고 쌓지 않기

⑥ 동작 낭비
   → 같은 작업을 매번 처음부터 다시 함
   → 해결: SOP + 스킬 파일 정비 (지금 하고 있는 것)

⑦ 결함 낭비
   → 수치 오류, 고객 이름 틀림, 링크 깨짐
   → 해결: ⚠️ 확인필요 표기 + 발송 전 진우님 검토 의무화
```

### 낭비 제거 우선순위 진단

부장이 프로세스 개선 요청 받을 때 먼저 물어볼 것:
```
"지금 가장 시간이 많이 걸리는 단계가 어디예요?"
"같은 작업을 반복하고 있는 게 있나요?"
"정보를 찾느라 시간 쓰는 일이 자주 있나요?"
```
→ 답변 기반으로 위 7가지 낭비 중 해당 항목 찾아 해결책 설계

## FRAMEWORK 3 — 제약이론 TOC (Eliyahu Goldratt)

**핵심 원칙:** "전체 속도는 가장 느린 병목이 결정한다. 병목만 개선하면 전체가 빨라진다."

### JW Financial 업무 흐름 병목 진단

```
[잠재 고객] → [상담 신청] → [1차 미팅] → [2차 미팅]
    → [포트폴리오 실행] → [온보딩] → [장기 고객]
    → [소개/레퍼럴] → [신규 잠재 고객]
```

각 단계별 병목 가능 지점:

```
상담 신청 단계:
  병목: 콘텐츠 → 신뢰 → 신청 전환율 낮음
  해결: marketing.md 콘텐츠 전략 강화

1차→2차 단계:
  병목: 미팅 사이 팔로업 없음
  해결: kakao-message.md 이탈방지 시퀀스 자동 트리거

2차→3차 단계:
  병목: 보험 생소함으로 이탈 (가장 큰 병목)
  해결: D+1/2/3 시퀀스 + 포트폴리오 가시화

포트폴리오 실행 단계:
  병목: 진우님이 직접 모든 자료 작성
  해결: 부장 에이전트 활용, 템플릿화

콘텐츠 생산 단계:
  병목: 아이디어 → 초안 → 발행까지 시간 과다
  해결: content-creation.md 원자→분자 재활용 전략

소개 단계:
  병목: 소개 요청 타이밍/방법 불명확
  해결: D+90 소개 요청 시퀀스 + Give 먼저 전략
```

### 병목 개선 작업 요청 형식

부장이 병목 개선안 만들 때 구조:
```
1. 현재 상태: [지금 어떻게 돌아가는가]
2. 병목 지점: [어디서 막히는가]
3. 원인 분석: [왜 막히는가]
4. 개선안: [무엇을 바꾸면 되는가]
5. 예상 효과: [얼마나 빨라지는가]
```

## 자동화 설계 원칙

새 자동화 파이프라인 설계 요청 시 부장이 따를 구조:

```
TRIGGER (언제 시작되는가)
  예: "2차완료 [고객명]" 메시지 수신 시

ACTION (무엇을 하는가)
  예: D+1/2/3 메시지 초안 생성 → 진우님 Telegram 전달

CONDITION (어떤 조건이 있는가)
  예: 고객별 포트폴리오 정보 필요 → Notion에서 조회

OUTPUT (무엇이 나오는가)
  예: 발송 준비된 메시지 3종 + ✅ 검토 후 발송 표기

HUMAN IN THE LOOP (사람이 개입하는 지점)
  예: 진우님 최종 확인 후 발송
```

### 현재 자동화 가능 vs 사람 필요 구분

```
자동화 가능:
  ✅ 모닝 브리핑 수집·요약
  ✅ 상담 인테이크 → Notion 저장 → 진우님 알림
  ✅ 날짜 기반 팔로업 리마인더 (D+7, D+30, D+90)
  ✅ 콘텐츠 초안 생성 (트리거 → 부장 → 검토 요청)
  ✅ 녹취 → 상담 요약 → 카카오 초안

사람(진우님) 필요:
  ⚠️ 고객 발송 최종 확인
  ⚠️ 포트폴리오 설계 (개인화 판단)
  ⚠️ 금융 수치 검증
  ⚠️ 외부 발행 (블로그, SNS 최종 업로드)
```

## 작업 요청 유형별 출력 형식

| 요청 | 출력 형식 |
|------|-----------|
| `[업무] 자동화 설계해줘` | TRIGGER→ACTION→CONDITION→OUTPUT→HUMAN 구조 |
| `[단계] 병목 찾아줘` | TOC 5단계 분석 (현재→병목→원인→개선→효과) |
| `주간 루틴 정리해줘` | GTD 주간 리뷰 체크리스트 |
| `노션 구조 짜줘` | DB 구조 + 뷰 설계안 |
| `[프로세스] SOP 만들어줘` | 단계별 SOP + 담당/도구/산출물 |
| `낭비 찾아줘` | 7가지 낭비 진단 + 우선 제거 항목 |
| `파이프라인 점검해줘` | 현재 파이프라인 현황 + 개선 제안 |

## 운영 원칙 요약

```
1. 시스템이 기억하게 — 머릿속 비우기 (GTD)
2. 가치 없는 것 제거 — 낭비 먼저 찾기 (도요타)
3. 병목 하나씩 — 전체 말고 막힌 곳만 (TOC)
4. 자동화 가능한 것은 자동화 — 진우님 시간은 고객에게
5. 사람이 필요한 것은 사람이 — 확인·판단·관계는 위임 불가
```
