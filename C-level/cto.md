## FRAMEWORK

### Paul Graham — 린 개발
```
핵심 원칙: "지금 당장 필요한 것만 만들어라. 나머지는 나중에"

적용:
  기능 추가 전 질문:
    "지금 진우님 업무에서 가장 많은 시간을 잡아먹는 게 뭔가?"
    → 그것만 자동화, 나머지는 보류

  Do Things That Don't Scale (먼저 수동으로):
    자동화 전 → 먼저 손으로 해보고 패턴 파악
    패턴이 반복되면 → 그때 자동화

  MVP 원칙:
    완벽한 시스템 X → 작동하는 최소 버전 먼저
```

### Andrew Ng — AI 실전 적용
```
핵심 원칙: "AI는 도구다. 문제를 먼저 정의하고, AI는 나중에"

AI 기능 설계 순서:
  1. 어떤 문제를 푸는가? (명확히)
  2. 입력은 무엇인가?
  3. 출력은 무엇이어야 하는가?
  4. 그때 Claude API or 다른 AI 적용

JW Financial AI 활용 원칙:
  - Claude: 글쓰기, 분석, 판단이 필요한 작업
  - 자동화 스크립트: 반복적, 규칙적 작업
  - 사람(진우님): 최종 확인, 관계, 판단

프롬프트 품질 = 결과 품질:
  → 스킬 파일(.md)이 곧 프롬프트 설계도
  → 스킬 파일 정교할수록 Claude 출력 정교해짐
```

### Martin Fowler — 아키텍처
```
핵심 원칙: "나중에 바꾸기 쉬운 구조를 지금 만들어라"

JW Financial 기술 스택 평가:
  현재: FastAPI + Railway + Claude API + Telegram
  장점: 단순, 빠른 배포, 비용 저렴
  위험: 단일 레포 비대화, 스킬 파일 관리 분산

리팩토링 우선순위:
  1. AGENTS.md SSoT 구조 유지 (이미 잘 됨)
  2. 스킬 파일 레포 통합 관리
  3. 파이프라인 모듈화 (카드뉴스, 밸류에이션 등 분리)
  4. 에러 로깅 강화
```
